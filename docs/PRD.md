# 커피 주문 시스템 PRD

- 문서 상태: 설계 확정
- 작성일: 2026-07-10
- 관련 문서: [아키텍처](./ARCHITECTURE.md), [ERD](./ERD.md), [API 명세](./API.md), [ADR 목록](./adr/README.md)

## 1. 배경

여러 서버 인스턴스에서 동시에 요청을 처리하더라도 포인트와 주문의 정합성이 깨지지 않는 커피 주문 시스템을 만든다. 사용자는 사전에 등록된 메뉴를 조회하고 포인트를 충전한 뒤, 포인트로 메뉴 1개를 주문·결제할 수 있다. 결제된 주문은 데이터 수집 플랫폼으로 신속하게 전달하며, 최근 주문 원본을 기준으로 인기 메뉴를 제공한다.

이 과제의 핵심은 기능 수가 아니라 다음 질문에 답할 수 있는 설계와 검증이다.

- 같은 사용자의 요청이 동시에 실행될 때 잔액이 어떻게 보호되는가?
- 네트워크 재시도가 중복 충전이나 중복 주문을 만들지 않는가?
- 주문 커밋과 외부 전송 사이의 이중 쓰기 문제를 어떻게 다루는가?
- 인기 메뉴의 기간, 상태, 동률 기준이 명확하고 집계 결과가 정확한가?
- 애플리케이션 인스턴스가 늘어나도 동일한 보장이 유지되는가?

## 2. 목표

### 2.1 제품 목표

1. 주문 가능한 메뉴의 ID, 이름, 가격을 조회한다.
2. 사용자 식별값과 충전 금액으로 포인트를 충전한다. `1원 = 1P`이다.
3. 사용자 식별값과 메뉴 ID로 주문과 포인트 결제를 한 번에 처리한다.
4. 결제 완료 주문을 데이터 수집 플랫폼으로 전달한다.
5. 조회 시각 기준 직전 168시간의 인기 메뉴 상위 3개를 정확히 제공한다.

### 2.2 품질 목표

- **일관성**: 주문, 포인트 차감, 포인트 원장, Outbox 이벤트는 하나의 MySQL 트랜잭션으로 커밋하거나 모두 롤백한다.
- **동시성 안전성**: 같은 사용자의 포인트 변경은 지갑 행의 비관적 락으로 직렬화한다.
- **다중 인스턴스 안전성**: 정합성 판단을 애플리케이션 메모리나 로컬 락에 의존하지 않는다.
- **멱등성**: 충전과 주문은 `Idempotency-Key`와 DB 유니크 제약으로 중복 효과를 방지한다.
- **전달 신뢰성**: 외부 플랫폼 장애가 주문 성공을 취소하지 않는다. Outbox는 중복 가능한 at-least-once 처리를 수행하고, 자동 재시도 한도 뒤에도 이벤트를 삭제하지 않고 격리해 수동 재처리할 수 있게 한다.
- **집계 정확성**: 인기 순위는 Redis나 비동기 파생 집계가 아니라 `PAID` 주문 원본에서 계산한다.
- **검증 가능성**: MySQL Testcontainers를 사용해 실제 락, 트랜잭션, 인덱스 조건에서 테스트한다.

## 3. 범위

### 3.1 V1 포함 범위

- 활성 메뉴 목록 조회
- 포인트 충전
- 메뉴 1개, 수량 1개의 주문·결제
- 결제 시점 메뉴 이름과 가격 스냅샷 보관
- 포인트 현재 잔액과 불변 원장 관리
- 충전·주문 요청 멱등 처리
- 주문 이벤트 Outbox 저장, 비동기 전송, 재시도와 실패 격리
- 최근 168시간 인기 메뉴 상위 3개 조회
- 기능, 예외, 동시성, 통합 테스트
- 다중 애플리케이션 인스턴스와 공유 MySQL을 전제로 한 설계

### 3.2 V1 제외 범위

- 회원가입, 로그인, 인증·인가
- 사용자 및 메뉴 등록·수정·삭제 API
- 장바구니, 복수 메뉴, 수량 변경
- 주문 취소·환불
- 포인트 사용 취소 및 만료
- 쿠폰, 할인, 복합 결제
- 관리자용 Outbox 재처리 API와 운영 UI
- 다국어, 다중 통화

사용자, 포인트 지갑, 메뉴는 Flyway 초기 데이터로 준비한다. 알 수 없는 사용자나 메뉴는 자동 생성하지 않는다.

## 4. 사용자 흐름

### 4.1 메뉴 조회

1. 사용자가 메뉴 목록을 요청한다.
2. 시스템은 현재 `ACTIVE` 상태인 메뉴만 반환한다.
3. Redis 도입 후에는 공유 캐시를 먼저 조회하고, 캐시 장애나 미스 시 MySQL에서 조회한다.

### 4.2 포인트 충전

1. 사용자가 고유한 `Idempotency-Key`와 양의 정수 충전액을 전송한다.
2. 시스템은 사용자를 확인하고 포인트 지갑 행을 잠근다.
3. 오버플로를 검증한 뒤 잔액과 포인트 원장을 함께 기록한다.
4. 같은 키와 같은 요청의 재전송에는 최초 완료 응답을 반환한다.

### 4.3 주문·결제

1. 사용자가 고유한 `Idempotency-Key`, 사용자 ID, 메뉴 ID를 전송한다.
2. 시스템은 주문 가능한 메뉴와 현재 가격을 확인한다.
3. 사용자의 포인트 지갑 행을 잠그고 잔액을 확인한다.
4. 잔액 차감, `PAID` 주문, 포인트 원장, Outbox 이벤트를 한 트랜잭션으로 저장한다.
5. 커밋 후 API는 주문 성공을 반환한다.
6. 별도 발행기가 Outbox 이벤트를 데이터 수집 플랫폼으로 전송한다.

### 4.4 인기 메뉴 조회

1. 사용자가 인기 메뉴 조회를 요청한다.
2. 시스템은 결제 주문 원본을 메뉴별로 집계한다.
3. [인기 메뉴 비즈니스 규칙](#64-인기-메뉴)에 따라 상위 메뉴를 반환한다.

## 6. 비즈니스 규칙

### 6.1 메뉴와 주문

- 한 주문은 메뉴 1개, 수량 1개만 가진다.
- 주문 요청은 가격을 받지 않는다. 서버가 메뉴의 현재 가격을 사용한다.
- 주문에는 메뉴 ID와 함께 결제 시점의 메뉴 이름, 단가, 결제 금액을 스냅샷으로 저장한다.
- `INACTIVE` 메뉴는 목록과 인기 추천에서 제외하며 주문할 수 없다.
- V1에는 취소와 환불이 없으므로 성공한 주문의 상태는 `PAID`이다.

### 6.2 포인트

- 포인트와 메뉴 가격은 소수점 없는 `BIGINT` 정수로 저장한다.
- 충전 금액은 `1P` 이상이어야 한다. 임의의 1회 충전 상한이나 잔액 상한은 두지 않는다.
- 정수 범위를 넘는 충전은 거절한다.
- 잔액은 음수가 될 수 없다.
- 원장의 `amount`는 양수로 저장하고 `CHARGE` 또는 `PAYMENT` 유형으로 방향을 구분한다.

### 6.3 멱등성

- 충전과 주문 요청은 1~128자의 `Idempotency-Key`를 필수로 보낸다.
- 멱등성 범위는 `(userId, operation, idempotencyKey)`이다.
- 요청 내용은 정규화한 뒤 SHA-256 해시로 비교한다.
- 같은 키와 같은 해시의 성공 응답은 최초 완료 상태 코드와 본문을 그대로 반환한다.
- 결정적 비즈니스 실패는 최초 상태 코드와 안정적인 오류 payload를 재생하되, 요청별 `timestamp`와 `traceId`는 현재 요청 값으로 새로 만든다.
- 같은 키와 다른 해시는 `409 Conflict`로 거절한다.
- `201` 성공뿐 아니라 실행이 시작된 뒤 확정된 메뉴 없음·비활성, 잔액 부족, 충전 오버플로 응답도 도메인 변경 없이 멱등 결과로 커밋한다.
- 구조 검증 실패, 사용자 없음, DB·락·서버의 일시적 실패는 멱등 결과로 저장하지 않는다.
- 저장된 비즈니스 실패의 조건을 해결한 뒤 다시 실행하려면 새 키를 사용한다.

### 6.4 인기 메뉴

- 조회 요청마다 주입받은 현재 시각을 `to`로 한 번 고정하고 `from = to - 168시간`을 계산한다.
- 집계 대상은 `paid_at`이 `[from, to)`인 `PAID` 주문 중 현재 `ACTIVE` 메뉴다.
- DB에는 UTC로 저장하고 API에는 RFC 3339 UTC 문자열로 반환한다.
- 순위는 주문 횟수 내림차순, 메뉴 ID 오름차순으로 결정한다.
- 주문된 메뉴가 3개 미만이면 존재하는 항목만 반환한다.

### 6.5 외부 이벤트

- 외부 전송 실패는 이미 커밋된 주문·결제를 롤백하지 않는다.
- 정상 상태에서는 주문 커밋 후 1초 이내에 첫 전송을 시도한다.
- 자동 처리의 전달 의미는 중복 가능한 `at-least-once`이며 수신자는 `eventId`로 중복을 제거한다.
- 재시도 가능한 실패는 지수 백오프로 최초 시도 후 최대 10회 재시도한다.
- 재시도 한도를 초과하면 이벤트를 `FAILED`로 격리한다. 운영자가 원인을 해결한 뒤 시도 횟수와 lease를 초기화해 새 재시도 주기로 전환할 수 있다.

## 7. 우선순위와 단계

### Phase 1 — 핵심 기능과 정합성

- Spring Boot 3.x, Java 21, JPA, MySQL 8.x 기반 단일 배포 애플리케이션
- 네 가지 필수 API
- 비관적 락, 포인트 원장, 멱등성, Transactional Outbox
- Mock HTTP 데이터 플랫폼 어댑터
- Flyway 초기 스키마·데이터
- JUnit 5와 Testcontainers MySQL 기반 테스트

### Phase 2 — 향후 인프라 확장 (제출 범위 제외)

- Outbox 발행 대상을 Kafka 토픽으로 교체
- 활성 메뉴 목록에 Redis Cache-Aside 적용
- Redis 장애 시 MySQL 폴백
- Kafka·Redis Testcontainers 통합 테스트

Kafka와 Redis는 정합성의 정본이 아니다. Kafka는 이벤트 전달 채널이고 Redis는 재생성 가능한 메뉴 조회 캐시다.

### 8.1 Phase 1

| 시나리오 | 기대 결과 |

| 잔액 10,000P, 가격 1,000P 메뉴에 동시 주문 20건 | 정확히 10건만 성공하고 잔액은 0P이며 주문·결제 원장·Outbox가 각각 10건이다. |
| 같은 충전 요청을 같은 키로 20회 동시 전송 | 충전 효과와 충전 원장은 1건이며 모든 성공 재생 응답의 결과가 같다. |

| 잔액 부족 주문 | 주문, 차감 원장, Outbox가 모두 생성되지 않는다. |
| 잔액 부족 주문을 같은 키로 재전송 | 최초 `409 INSUFFICIENT_POINTS` 응답을 재생하며, 충전 후 재실행하려면 새 키를 사용한다. |
| Outbox 저장 단계에서 예외 발생 | 주문과 포인트 차감이 모두 롤백된다. |
| 외부 플랫폼이 타임아웃 | 주문은 성공하고 이벤트는 재시도 대상이 된다. |
| 외부 플랫폼이 계속 실패 | 최초 시도 후 최대 10회 재시도하고 `FAILED`로 격리한다. |
| 기간 경계에 주문 존재 | `from`과 같은 시각은 포함하고 `to`와 같은 시각은 제외한다. |
| 인기 횟수 동률 | 메뉴 ID가 작은 항목이 먼저 나온다. |

## 9. 완료 정의

### 9.1 Phase 1 완료

- 네 가지 필수 API와 명시된 예외 응답이 구현되어 있다.
- 모든 스키마 변경과 초기 데이터가 Flyway로 재현된다.
- 단위, 통합, 동시성, 외부 연동 실패 테스트가 통과한다.
- 로컬 JVM 락이나 인스턴스 로컬 정합성 상태에 의존하지 않는다.
- 네 핵심 문서와 ADR이 실제 구현과 일치한다.

### 9.2 과제 제출 완료

- [Phase 1 완료](#91-phase-1-완료)의 기능·정합성·테스트 조건을 충족한다.
- Mock HTTP 수신자와 Transactional Outbox로 외부 주문 이벤트 전달 요구를 충족하고, 전송 실패가 주문·결제를 롤백하지 않음을 검증한다.
- 최종 `README.md`에서 설계 의도, 문제 해결 전략, 기술 선택, ERD, API 문서로 연결한다.
- 기능 단위 커밋과 원격 push 이력이 남아 있다.

Kafka 발행과 Redis 메뉴 캐시는 제출 이후의 확장 방향이다. 두 인프라를 과제 완료 조건으로 두지 않으며, 도입 시에는 부하·운영 조건과 장애·폴백 통합 테스트를 별도 ADR과 구현 변경에서 확정한다.
