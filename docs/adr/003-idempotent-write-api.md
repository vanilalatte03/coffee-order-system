# ADR-003: 충전과 주문 API에 DB 기반 멱등성 적용

- 상태: 채택됨
- 날짜: 2026-07-10
- 관련: [`PRD` 멱등성](../PRD.md#63-멱등성), [`ARCHITECTURE` 멱등 처리](../ARCHITECTURE.md#53-멱등성-처리), [`API` 공통 규칙](../API.md#02-쓰기-요청-멱등성), [`ERD` 멱등 요청](../ERD.md#36-idempotency_requests)

## 맥락

클라이언트는 응답 유실, timeout, 사용자 재시도로 같은 충전이나 주문을 다시 보낼 수 있다. 포인트 지갑의 비관적 락은 동시에 실행되는 잔액 변경을 직렬화하지만, 첫 요청이 커밋된 뒤 순차적으로 도착한 같은 요청까지 구분하지 못한다. 이 경우 중복 충전과 중복 결제가 발생할 수 있다.

## 검토한 대안

- **별도 보호 없음** — API는 단순하지만 네트워크 재시도가 중복 재무 효과를 만든다.
- **애플리케이션 로컬 캐시** — 빠르지만 인스턴스별 값이 다르고 재시작 시 기록이 사라진다.
- **Redis 멱등 키** — 여러 인스턴스가 공유할 수 있지만 DB 커밋과 Redis 기록 사이에 이중 쓰기 문제가 생긴다.
- **도메인 테이블별 요청 키 컬럼** — 강한 DB 제약은 가능하지만 충전 원장과 주문에 로직과 응답 저장 방식이 중복된다.
- **공통 DB 멱등 요청 테이블** — 연산 범위, 요청 해시, 최초 완료 응답을 한 모델로 저장하고 도메인 처리 결과와 함께 커밋한다.

## 결정

포인트 충전과 주문·결제에 `Idempotency-Key` 헤더를 필수로 적용한다. DB의 `(user_id, operation, idempotency_key)` 유니크 제약으로 승자를 하나만 허용한다. 요청 Path와 body를 정규화한 SHA-256 해시를 저장한다.

- 같은 키·같은 해시: 최초 완료 HTTP 상태와 본문을 그대로 재생한다.
- 같은 키·다른 해시: `409 IDEMPOTENCY_KEY_REUSED`로 거절한다.
- 성공과 실행 후 확정된 결정적 비즈니스 실패를 저장한다.
- 구조 검증, 사용자 없음, DB·락·서버의 일시적 실패는 저장하지 않는다.
- V1에서는 완료 기록을 자동 만료시키지 않는다.

동시에 같은 키가 들어와 유니크 제약이 충돌하면 실패한 트랜잭션 밖의 새 읽기 트랜잭션에서 승자의 완료 기록을 조회한다.

## 결과

- timeout 후 같은 키를 재전송해도 충전·주문 효과가 한 번만 발생한다.
- 최초 응답 당시의 잔액·주문 ID 또는 비즈니스 오류를 정확히 재생하도록 응답 JSON 스냅샷을 저장한다.
- 키 저장 공간과 응답 보존 비용이 증가한다.
- 유니크 충돌과 재조회는 트랜잭션 프록시 경계를 분리해 구현해야 한다.
- 키 보존 기간을 나중에 도입하면 키 재사용 계약과 정리 작업을 새 ADR로 결정한다.
- 저장된 잔액 부족과 같은 조건을 해결한 뒤 다시 실행하려면 새 키를 사용해야 한다.
