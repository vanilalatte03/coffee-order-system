# ADR-001: 모듈러 모놀리스와 공유 MySQL을 시스템 기준으로 사용

- 상태: 채택됨
- 날짜: 2026-07-10
- 관련: [`PRD`](../PRD.md), [`ARCHITECTURE`](../ARCHITECTURE.md), [`ERD`](../ERD.md)

## 맥락

과제는 여러 서버 인스턴스에서도 포인트와 주문의 정합성을 보장해야 한다. 동시에 정해진 기간 안에 네 가지 API, 동시성 제어, 외부 전송, 인기 집계와 테스트를 완성해야 한다. 초기부터 서비스를 분리하거나 Kafka·Redis를 핵심 쓰기 경로에 넣으면 운영 요소와 분산 실패 지점이 늘어난다. 반대로 단일 인스턴스와 로컬 상태에 의존하면 도전 요구사항을 만족할 수 없다.

## 검토한 대안

- **단일 인스턴스와 인메모리 상태** — 구현은 가장 단순하지만 프로세스 재시작과 수평 확장 시 락, 멱등성, 이벤트가 분리된다.
- **초기부터 마이크로서비스와 Kafka·Redis 도입** — 기능별 확장성은 높지만 분산 트랜잭션, 배포, 로컬 개발과 테스트 비용이 과제 핵심보다 커진다.
- **모듈러 모놀리스와 공유 MySQL** — 배포 단위는 하나지만 기능 경계를 코드로 분리하고, 여러 Stateless 인스턴스가 하나의 DB 정합성 기준을 공유한다.

## 결정

모듈러 모놀리스를 사용해 메뉴, 포인트, 주문, 인기 조회, 멱등성, Outbox를 같은 배포 단위 안의 독립 기능 경계로 나눈다. 애플리케이션은 Stateless하게 수평 확장하고 모든 인스턴스는 같은 MySQL을 사용한다. 구체적인 언어·프레임워크와 버전은 아키텍처 문서를 정본으로 둔다.

MySQL을 핵심 상태의 유일한 정본으로 둔다. 프로세스 로컬 락, 인메모리 큐, Redis 캐시를 정합성 근거로 사용하지 않는다. Kafka와 Redis의 적용 순서는 PRD, 구체적인 역할은 관련 ADR을 정본으로 둔다.

## 결과

- 여러 인스턴스가 DB 행 락, 트랜잭션, 유니크 제약을 공유한다.
- 단일 트랜잭션으로 주문, 포인트, Outbox를 처리할 수 있다.
- 모듈별 포트 뒤에서 Mock HTTP를 Kafka로, DB 조회를 Redis 캐시로 확장할 수 있어야 한다.
- MySQL 장애가 쓰기 가용성의 단일 실패 지점이며 높은 부하에서는 공유 DB가 병목이 될 수 있다.
- 현재 과제에서는 운영 복잡도보다 정확성과 설명 가능성을 우선한다.
- 서비스 분리는 실제 독립 확장 요구와 부하 데이터가 생길 때 새 ADR로 결정한다.
