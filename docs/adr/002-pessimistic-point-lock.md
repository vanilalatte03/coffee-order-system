# ADR-002: 포인트 지갑에 DB 비관적 락 적용

- 상태: 채택됨
- 날짜: 2026-07-10
- 관련: [`PRD` 포인트 규칙](../PRD.md#62-포인트), [`ARCHITECTURE` 정합성 전략](../ARCHITECTURE.md#5-정합성과-동시성-전략), [`ERD` 지갑](../ERD.md#33-point_wallets)

## 맥락

같은 사용자의 충전과 주문이 동시에 실행되면 두 요청이 같은 이전 잔액을 읽어 갱신을 덮어쓰거나 잔액이 음수가 될 수 있다. 여러 애플리케이션 인스턴스가 요청을 처리하므로 JVM `synchronized`나 로컬 락으로는 보호할 수 없다. 잔액 변경과 포인트 원장, 주문, Outbox는 하나의 명확한 트랜잭션 흐름으로 설명하고 테스트할 수 있어야 한다.

## 검토한 대안

- **애플리케이션 로컬 락** — 단일 프로세스에서는 쉽지만 인스턴스가 늘어나면 같은 사용자를 동시에 잠글 수 없다.
- **JPA 낙관적 락과 재시도** — 충돌이 드물 때 유리하지만 포인트처럼 동일 사용자 쓰기가 연속될 수 있는 경로에서 충돌 응답 또는 재시도 정책이 추가된다.
- **조건부 원자 UPDATE** — `balance >= amount` 조건으로 차감은 효율적이지만 변경 직후 잔액을 원장·응답에 일관되게 연결하고 충전·주문 흐름을 동일하게 표현하는 코드가 복잡해진다.
- **Redis 분산 락** — DB 밖의 lease, 만료, 장애 정책이 추가되며 DB 트랜잭션과 락 소유권이 분리된다.
- **MySQL 비관적 락** — 사용자 지갑 행을 `SELECT ... FOR UPDATE`로 잠가 트랜잭션 안의 포인트 변경을 직렬화한다.

## 결정

포인트 충전과 주문·결제는 `point_wallets.user_id` 행을 비관적 쓰기 락으로 조회한다. 잔액 검증·변경, 포인트 원장과 관련 주문·Outbox 저장은 락을 가진 같은 트랜잭션에서 수행한다.

모든 포인트 쓰기는 `입력·멱등성 → 지갑 락 → 도메인 쓰기 → 커밋` 순서를 따른다. 외부 HTTP·Kafka·Redis 호출은 지갑 락을 가진 트랜잭션 안에서 수행하지 않는다.

## 결과

- 다중 인스턴스에서도 같은 사용자의 포인트 변경이 DB에서 직렬화된다.
- 잔액 부족 검증과 차감 사이의 경쟁 조건이 사라진다.
- 서로 다른 사용자는 다른 행을 잠그므로 병렬 처리할 수 있다.
- 핫 사용자에게 요청이 몰리면 락 대기와 timeout이 발생할 수 있다.
- 트랜잭션을 짧게 유지하고 락 획득 순서를 통일해야 한다.
- 락 timeout은 `503 CONCURRENCY_TIMEOUT`으로 변환해 같은 멱등 키 재시도를 허용한다.
- H2가 아닌 MySQL Testcontainers에서 서로 다른 커넥션의 동시 요청으로 검증한다.
