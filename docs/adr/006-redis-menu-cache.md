# ADR-006: Redis를 활성 메뉴 목록 Cache-Aside로 사용

- 상태: 채택됨
- 날짜: 2026-07-10
- 관련: [`PRD` Phase 2](../PRD.md#phase-2--인프라-확장), [`ARCHITECTURE` Redis 캐시](../ARCHITECTURE.md#9-redis-메뉴-캐시), [`API` 메뉴 목록](../API.md#1-메뉴-목록-조회)

## 맥락

메뉴 목록은 모든 사용자가 반복해서 조회하며 쓰기보다 읽기가 훨씬 많다. 여러 애플리케이션 인스턴스에서 같은 데이터를 반복 조회하면 불필요한 MySQL 부하가 생길 수 있다. 하지만 Redis를 포인트, 멱등성 또는 인기 메뉴의 정본으로 사용하면 DB 트랜잭션과 분리된 정합성 문제를 새로 만든다.

## 검토한 대안

- **캐시 없음** — 장애 지점이 적고 항상 최신이지만 모든 메뉴 조회가 MySQL에 도달한다.
- **인스턴스 로컬 캐시** — 도입은 쉽지만 인스턴스마다 값과 만료 시점이 달라지고 배포 시 초기화된다.
- **Redis Cache-Aside** — 여러 인스턴스가 캐시를 공유하고 TTL을 적용할 수 있지만 별도 인프라와 장애 폴백이 필요하다.
- **Redis에 인기 순위·포인트도 저장** — 빠른 조회가 가능하지만 “정확한 주문 횟수”와 포인트 트랜잭션의 정본이 분리된다.

## 결정

핵심 API와 정합성 구현이 완료된 Phase 2에 Redis를 도입한다. Redis는 `ACTIVE` 메뉴 목록만 Cache-Aside 방식으로 캐시한다.

- 캐시 미스: MySQL 조회 후 Redis 저장
- Redis timeout·장애: MySQL로 폴백
- 주문 시 메뉴 상태와 가격: 캐시가 아니라 MySQL에서 다시 확인
- 인기 메뉴, 포인트, 멱등성: Redis를 정합성 기준으로 사용하지 않음

캐시 키, TTL, jitter와 빈 결과 캐싱 같은 되돌리기 쉬운 운영 설정은 아키텍처 문서를 정본으로 둔다.

## 결과

- 여러 인스턴스의 반복 메뉴 조회가 같은 캐시를 공유한다.
- Redis가 중단되어도 메뉴 API가 MySQL로 계속 동작한다.
- Redis 데이터는 삭제해도 MySQL에서 재생성할 수 있다.
- TTL 동안 메뉴 변경 반영이 늦을 수 있으나 V1에는 메뉴 관리 API가 없다.
- 향후 메뉴 관리 API를 추가하면 커밋 후 캐시 무효화와 stampede 제어를 함께 설계해야 한다.
- 인기 메뉴의 정확성은 이 캐시의 상태와 무관하다.
