# ADR-005: 인기 메뉴를 MySQL 주문 원본에서 직접 집계

- 상태: 채택됨
- 날짜: 2026-07-10
- 관련: [`PRD` 인기 메뉴](../PRD.md#64-인기-메뉴), [`ARCHITECTURE` 인기 집계](../ARCHITECTURE.md#8-인기-메뉴-집계), [`API` 인기 메뉴](../API.md#4-인기-메뉴-조회), [`ERD` 주문](../ERD.md#35-orders)

## 맥락

최근 7일간 인기 메뉴 3개와 메뉴별 주문 횟수는 정확해야 한다. Kafka 소비자나 Redis 카운터처럼 비동기로 갱신되는 파생 데이터는 장애·재처리·지연 중 주문 원본과 달라질 수 있다. 반면 매 요청마다 주문 원본을 집계하면 데이터가 커질수록 DB 비용이 증가한다. 초기 과제 범위에서 정확성과 성능 최적화의 우선순위를 정해야 한다.

## 검토한 대안

- **Kafka 소비 + Redis Sorted Set** — 조회는 빠르지만 소비 지연, 중복 이벤트, 기간 만료 처리로 원본과 일시적으로 다를 수 있다.
- **일별 집계 테이블** — 조회 비용을 줄일 수 있지만 집계 갱신, 원본 대사, 현재 시각에서 168시간 경계를 잘라내는 로직이 추가된다.
- **Redis 결과 캐시** — 단순하지만 캐시 TTL 동안 새 주문과 기간 만료가 반영되지 않는다.
- **MySQL 주문 원본 직접 집계** — 가장 단순하고 커밋된 주문을 정확히 반영하지만 주문량이 커지면 집계 부하가 증가한다.

## 결정

V1 인기 메뉴는 MySQL `orders` 원본에서 직접 집계한다.

- 조회 시 한 번 고정한 UTC `to`를 사용한다.
- 범위는 `paid_at >= to - 168시간 AND paid_at < to`이다.
- 주문 상태는 `PAID`, 현재 메뉴 상태는 `ACTIVE`만 포함한다.
- `COUNT(*) DESC`, `menu_id ASC`, `LIMIT 3`으로 결정한다.
- Redis와 Kafka의 파생 카운터를 사용하지 않는다.
- `(status, paid_at, menu_id)` 인덱스로 기간·상태 필터를 지원한다.

## 결과

- 주문 트랜잭션이 커밋되는 즉시 다음 조회의 집계 대상이 된다.
- 멱등 재생, 실패·롤백 주문은 별도 보정 없이 중복 집계되지 않는다.
- 캐시 정합성이나 이벤트 소비 지연을 인기 횟수의 오류 원인에서 제거한다.
- 데이터가 커질수록 범위 스캔과 그룹 집계 비용이 증가한다.
- 실제 실행 계획과 응답 지연을 관측하고 목표를 넘을 때만 일별 집계 또는 별도 분석 저장소를 검토한다.
- 사전 집계를 도입할 때는 지연 허용 범위, 원본 대사, 재처리 정책을 새 ADR로 결정한다.
