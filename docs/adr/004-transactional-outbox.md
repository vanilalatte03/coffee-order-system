# ADR-004: 주문 이벤트를 Transactional Outbox로 전달

- 상태: 채택됨
- 날짜: 2026-07-10
- 관련: [`PRD` 외부 이벤트](../PRD.md#65-외부-이벤트), [`ARCHITECTURE` 이벤트 전달](../ARCHITECTURE.md#7-외부-이벤트-전달), [`API` 이벤트 계약](../API.md#5-데이터-플랫폼-이벤트-계약), [`ERD` Outbox](../ERD.md#37-outbox_events)

## 맥락

주문·포인트 결제는 MySQL에 저장되고 주문 내역은 별도의 데이터 수집 플랫폼으로 전달되어야 한다. DB 커밋과 외부 호출은 하나의 로컬 트랜잭션으로 묶을 수 없다. DB만 커밋되거나 외부 전송만 성공하는 이중 쓰기 문제를 해결해야 하며, 외부 플랫폼 장애가 고객 주문을 실패시키지 않아야 한다.

## 검토한 대안

- **DB 트랜잭션 안에서 외부 HTTP 호출** — 즉시 결과를 알 수 있지만 느리거나 장애 난 외부 시스템이 포인트 지갑 락과 DB 커넥션을 오래 점유한다. 외부 성공 후 DB 롤백도 해결하지 못한다.
- **커밋 후 인메모리 비동기 작업** — 응답은 빠르지만 커밋과 작업 등록 사이에 프로세스가 종료되면 이벤트를 잃는다.
- **분산 트랜잭션·2PC** — 원자성은 강하지만 데이터 플랫폼·Kafka까지 프로토콜 지원과 큰 운영 복잡도가 필요하다.
- **Transactional Outbox** — 주문과 이벤트를 같은 DB 트랜잭션으로 저장하고 별도 작업자가 내구성 있게 발행한다. 중복 전송 가능성을 수용해야 한다.

## 결정

주문, 포인트 차감, 결제 원장, `ORDER_PAID` Outbox 이벤트를 하나의 MySQL 트랜잭션으로 저장한다. 외부 호출은 커밋 후 별도 작업자가 수행한다.

- Phase 1 발행 어댑터: Mock Data Platform HTTP API
- Phase 2 발행 어댑터: Kafka `coffee.order.paid.v1`
- 정상 첫 전송 목표: 주문 커밋 후 1초 이내
- 다중 작업자 선점: 짧은 `FOR UPDATE SKIP LOCKED` 트랜잭션과 `PROCESSING` lease
- 전달 의미: 중복 가능한 durable at-least-once 처리
- 중복 제거 키: UUID `eventId`
- 재시도: 한 자동 처리 주기에서 최초 dispatch 1회 이후 지수 백오프로 최대 10회 재선점, 정상 경로의 외부 호출 최대 11회
- 한도 도달 crash 복구: `attempt_count = 11`인 `PROCESSING` 작업자가 선점 커밋 후 종료하면 lease 만료 뒤 12회째 선점이나 외부 호출 없이 짧은 트랜잭션에서 `FAILED`로 전이하고 claim token과 lease를 정리
- 최종 실패: `FAILED` 격리 후 시도 횟수·claim token·lease를 초기화하는 새 주기의 수동 재처리

네트워크 호출 동안 DB 락을 잡지 않는다. 작업자는 먼저 상태와 lease를 커밋하고 호출하며, lease 만료 시 한도가 남은 행만 횟수를 증가시켜 회수한다. 이미 `attempt_count = 11`이면 DB CHECK를 넘는 12회째 선점을 시도하지 않고 외부 호출 없이 `FAILED`로 격리한다.

## 결과

- Outbox 저장 실패 시 주문과 포인트 차감도 함께 롤백된다.
- 외부 플랫폼 장애는 이미 커밋된 주문을 취소하지 않는다.
- 프로세스 종료 뒤에도 DB에 남은 이벤트를 다시 처리할 수 있다.
- 외부 수신 성공 후 `PUBLISHED` 기록 전에 종료되면 중복 전송될 수 있으므로 수신자의 `eventId` 멱등 처리가 필수다.
- 자동 재시도 한도 후에도 이벤트를 삭제하지 않으며 운영 재처리가 필요하다.
- Phase 2에서 Kafka를 도입해도 Outbox는 유지하고 발행 어댑터만 교체한다.
- Kafka 파티션, 보존, 스키마 관리의 세부 선택은 실제 도입 시 별도 ADR로 결정한다.
